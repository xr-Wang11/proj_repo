# LaTeX 计算器项目蓝图

## 1. 项目目标

本项目第一阶段要实现一个以命令行为入口的数学计算器核心，后续无缝扩展到 HTML 网页 UI。系统必须支持：

- 表达式解析与计算
- LaTeX 公式显示
- 复数计算
- 相量值与直角坐标形式互转
- 后续扩展为“将公式保存为函数并复用”的能力

第一阶段不直接实现网页 UI，也不实现用户函数持久化，只保留清晰接口。

## 2. 核心设计原则

### 2.1 AST 是唯一事实来源

所有输入表达式必须先被解析为 AST，再基于同一棵 AST：

- 进行数值计算
- 生成 LaTeX
- 生成命令行文本输出
- 未来保存为用户函数体

禁止分别维护“显示逻辑”和“计算逻辑”的两套表达式系统。

### 2.2 核心引擎与界面分离

项目结构分为两层：

- 核心引擎：解析、求值、格式化、错误系统、函数注册
- 接入层：CLI 和未来的网页 UI

核心引擎不能依赖命令行实现细节，也不能依赖 DOM。

### 2.3 复数是一等公民

内部数值系统默认支持复数。实数只是虚部为 0 的复数特例。相量是复数的一种输入/输出形式，而不是单独数学体系。

### 2.4 先收敛语法，再增加语法糖

第一阶段优先保证解析稳定，刻意限制高歧义语法：

- 支持 `3 + 4*i`
- 支持 `rect(3,4)` 和 `polar(5,30,deg)`
- 暂不支持隐式乘法
- 暂不支持 `5∠30` 这类特殊字符输入

## 3. 阶段范围

### 3.1 第一阶段必须实现

- 命令行 REPL 入口
- 基本四则运算和幂
- 括号与函数调用
- 常量 `pi`、`e`、`i`
- 复数四则运算
- 相量/直角坐标双向转换
- 表达式 LaTeX 输出
- 结果文本输出
- 面向未来用户函数的接口占位

### 3.2 第一阶段暂不实现

- 网页 UI
- 用户函数落盘存储
- 隐式乘法
- 多行脚本
- 矩阵
- 单位制计算
- 特殊输入语法如 `5∠30`

## 5. 统一数据模型

该模块必须最先定义，其他模块全部依赖它。

### 5.1 Token

用途：词法分析输出。

字段要求：

- `type`: token 类型
- `value`: 原始文本
- `start`: 起始位置
- `end`: 结束位置

建议类型：

- `NUMBER`
- `IDENTIFIER`
- `OPERATOR`
- `LPAREN`
- `RPAREN`
- `COMMA`
- `EOF`

### 5.2 ASTNode

用途：语法分析输出，系统统一表达式表示。

所有节点共有字段：

- `kind`
- `span`

第一阶段节点种类建议：

- `NumberLiteral`
- `Identifier`
- `UnaryExpression`
- `BinaryExpression`
- `FunctionCall`
- `ConversionExpression`

说明：

- `ConversionExpression` 可选。如果实现上更简单，也可以统一先用 `FunctionCall` 表示转换函数。

### 5.3 ComplexValue

用途：统一内部数值表示。

字段要求：

- `re`
- `im`

规则：

- 实数也是复数
- 允许内部统一按复数对象处理
- 需要定义数值容差，例如 `1e-12`

### 5.4 PhasorValue

用途：相量输出或相量输入中间表示。

字段要求：

- `magnitude`
- `angle`
- `angleUnit`

规则：

- `angleUnit` 仅允许 `deg` 或 `rad`
- 相量最终必须可转换为 `ComplexValue`

### 5.5 EvaluationResult

用途：求值器统一输出。

字段要求：

- `value`
- `displayType`
- `text`
- `latex`

说明：

- `value` 存内部数学值
- `displayType` 标记建议显示成实数、直角复数、相量等
- `text` 给 CLI
- `latex` 给未来网页 UI

### 5.6 FunctionDefinition

用途：内建函数和未来用户函数的统一结构。

字段建议：

- `name`
- `parameters`
- `bodyAst`
- `metadata`

说明：

- 第一阶段内建函数可以没有 `bodyAst`，而是提供执行器
- 未来用户函数以 `bodyAst` 为核心

## 6. 模块划分

## 6.1 模块 A：词法分析器 Tokenizer

职责：

- 将输入字符串切分为 token 序列
- 忽略空白字符
- 记录字符位置
- 提前发现非法字符

输入：

- 原始表达式字符串

输出：

- `Token[]`

必须支持：

- 整数
- 小数
- 科学计数法
- 标识符
- 运算符 `+ - * / ^`
- 括号
- 逗号

约束：

- 不处理数学优先级
- 不直接把 `-3` 作为负数字面量处理
- `i` 先作为标识符，交由语法层解释

建议不支持：

- 隐式乘法
- 特殊字符相量语法

完成标准：

- 任意支持表达式都能稳定产出 token
- 非法字符错误包含精确位置

## 6.2 模块 B：语法分析器 Parser

职责：

- 将 token 序列转换为 AST
- 处理优先级与结合性
- 识别函数调用和括号分组

输入：

- `Token[]`

输出：

- `ASTNode`

建议优先级：

1. 函数调用与括号
2. 幂 `^`
3. 一元 `+ -`
4. 乘除 `* /`
5. 加减 `+ -`

建议支持的表达式：

- `3 + 4*i`
- `(1+2*i) * (3-4*i)`
- `sin(pi/4)`
- `rect(3,4)`
- `polar(5,30,deg)`
- `to_polar(3+4*i,deg)`

关键规则：

- `^` 右结合
- 函数调用必须显式写括号
- 第一阶段不支持隐式乘法

错误要求：

- 缺括号时给出位置
- 连续非法运算符时给出位置
- 缺参数时给出当前参数槽位

完成标准：

- 支持表达式全部生成稳定 AST
- 错误具备可读信息与位置

## 6.3 模块 C：数学值系统

职责：

- 实现复数运算
- 实现相量与复数转换
- 定义数值规范化规则

输入：

- 数值对象

输出：

- 数值对象

必须支持：

- 加减乘除
- 幂
- 共轭
- 模长
- 相角
- 实部/虚部读取
- `rect -> complex`
- `polar -> complex`
- `complex -> polar`

角度规则：

- 内部统一使用弧度
- `deg` 只在输入输出时转换

实现要求：

- 统一处理近似 0
- 明确 NaN 和 Infinity 策略
- 显示层不能反向污染内部值系统

完成标准：

- 复数和相量转换结果稳定
- 误差控制可预期

## 6.4 模块 D：内建函数注册中心

职责：

- 注册和查找内建函数
- 为未来用户函数留统一入口

建议接口：

- `registerBuiltin(definition)`
- `getFunction(name)`
- `hasFunction(name)`
- `listFunctions()`
- `registerUserFunction(definition)`

第一阶段建议内建函数：

- `sin`
- `cos`
- `tan`
- `sqrt`
- `exp`
- `log`
- `abs`
- `arg`
- `re`
- `im`
- `conj`
- `rect`
- `polar`
- `to_rect`
- `to_polar`

元信息建议：

- 参数个数
- 参数名
- 是否支持复数
- 是否影响推荐显示形式

完成标准：

- 新增一个内建函数不需要修改求值器主流程

## 6.5 模块 E：求值器 Evaluator

职责：

- 遍历 AST
- 结合上下文环境进行求值
- 调用函数注册中心
- 输出统一结果对象

输入：

- `ASTNode`
- `EvaluationContext`

输出：

- `EvaluationResult`

`EvaluationContext` 至少包含：

- 常量表
- 变量表
- 函数表
- 全局选项

常量表至少支持：

- `pi`
- `e`
- `i`

全局选项建议：

- 默认角度单位
- 默认输出模式
- 小数精度

行为要求：

- 标识符按“变量优先、常量其次”查找
- 函数调用统一走注册中心
- `to_polar` 返回复数值，但推荐显示为相量
- 所有异常转换成结构化错误

完成标准：

- 支持表达式均可稳定求值
- 复数与相量显示意图不混乱

## 6.6 模块 F：LaTeX 渲染器

职责：

- 将 AST 转为表达式 LaTeX
- 将结果对象转为结果 LaTeX

输入：

- `ASTNode`
- `EvaluationResult`

输出：

- LaTeX 字符串

规则建议：

- 除法优先转 `\frac{}{}`
- 幂转 `a^{b}`
- 函数转 `\sin(x)` 或 `\operatorname{name}(...)`
- 复数结果渲染为 `a + bi`
- 相量结果渲染为 `r \angle \theta^\circ`

要求：

- 根据优先级自动补括号
- 正确处理负号与纯虚数
- 科学计数法有单独格式规则

完成标准：

- 同一 AST 可稳定产出统一 LaTeX
- 后续 KaTeX 或 MathJax 可以直接消费

## 6.7 模块 G：文本格式化器

职责：

- 将求值结果转为 CLI 可读文本
- 根据显示模式切换输出直角式或相量式

输入：

- `EvaluationResult`

输出：

- 文本字符串

建议模式：

- `plain`
- `polar`
- `debug`

规则要求：

- 复数输出如 `3 + 4i`
- 相量输出如 `5 ∠ 53.13 deg`
- 纯实数避免无意义长尾
- 错误输出包含提示信息

完成标准：

- CLI 不依赖 LaTeX 也可清晰展示结果

## 6.8 模块 H：错误系统

职责：

- 统一所有模块错误结构

错误种类建议：

- `LexError`
- `ParseError`
- `NameError`
- `ArityError`
- `MathDomainError`
- `ConversionError`
- `UnsupportedFeatureError`

每个错误至少包含：

- `code`
- `message`
- `position`
- `context`
- `hint`

完成标准：

- 不出现裸字符串异常
- CLI 和未来网页都能消费同一错误对象

## 6.9 模块 I：CLI 交互层

职责：

- 提供第一阶段用户入口
- 管理 REPL 循环
- 调用核心引擎

建议命令：

- `:help`
- `:quit`
- `:mode rect`
- `:mode polar`
- `:angle deg`
- `:angle rad`
- `:functions`
- `:history`

规则：

- 以 `:` 开头的输入为命令
- 其他输入视为表达式
- 每次计算至少输出文本结果
- 可额外输出 LaTeX 供调试

完成标准：

- 用户无需网页即可完成第一阶段目标计算

## 6.10 模块 J：网页输入协议预留

职责：

- 规划未来 HTML UI 的输入方式
- 明确键盘输入和按钮输入分工

你的约束必须落实为协议：

- 数值输入使用键盘
- 运算符和函数输入使用点击按钮

需要定义的前端状态：

- `rawInput`
- `cursorPosition`
- `parsedAst`
- `latexExpression`
- `evaluationResult`
- `displayMode`
- `angleUnit`

按钮插入策略必须提前明确：

- 运算符在光标位置插入
- 函数按钮插入模板，如 `sin()`
- 光标自动定位到括号内部

完成标准：

- 后续做网页 UI 时不需要重新设计输入协议

## 6.11 模块 K：用户函数接口占位

职责：

- 为“保存公式为函数”预留未来接入点

第一阶段只做接口，不做持久化。

未来目标能力：

- 保存函数名
- 保存参数列表
- 保存函数体 AST
- 调用时传入变量值求值

未来工作流约定：

1. 用户输入函数名
2. 用户输入参数列表
3. 用户输入公式字符串
4. 系统解析为 `bodyAst`
5. 求值时以局部变量上下文执行

需要提前确定的限制：

- 不允许覆盖内建函数
- 不允许参数名与常量名冲突
- 第一阶段默认不支持递归

完成标准：

- 后续实现用户函数时无需重构求值器主架构

## 6.12 模块 L：测试体系

职责：

- 为各核心模块建立最小回归集

测试分类：

- 词法测试
- 语法测试
- 求值测试
- 转换测试
- 显示测试

第一阶段必须覆盖的关键用例：

- `3 + 4*i`
- `(1+2*i) * (3-4*i)`
- `abs(3+4*i)`
- `arg(1+i)`
- `rect(3,4)`
- `to_polar(rect(3,4),deg)`
- `polar(5,53.1301023542,deg)`
- `to_rect(polar(5,53.1301023542,deg))`
- `i^2`
- 错误输入如 `2 + * 3`

测试原则：

- 数值比较采用容差
- 错误测试验证类型和位置
- LaTeX 测试验证格式输出

## 7. 推荐实现顺序

建议按照依赖顺序推进：

1. 统一数据模型

3. Parser
4. 数学值系统
5. 内建函数注册中心
6. Evaluator
7. 文本格式化器
8. LaTeX 渲染器
9. 错误系统统一收口
10. CLI
11. 网页输入协议预留
12. 用户函数接口占位
13. 测试补齐

## 8. 第一阶段建议支持的表达式集合

建议先将语法范围收敛到以下集合：

- `+ - * / ^`
- `()`
- `pi e i`
- `sin cos tan sqrt exp log`
- `abs arg re im conj`
- `rect(x,y)`
- `polar(r,theta,deg|rad)`
- `to_rect(z)`
- `to_polar(z,deg|rad)`

推荐输入示例：

```text
3 + 4*i
(1+2*i) * (3-4*i)
abs(3+4*i)
rect(3,4)
polar(5,30,deg)
to_polar(3+4*i,deg)
to_rect(polar(5,30,deg))
```

## 9. 当前阶段的实现边界

为了保证第一版稳定，以下内容应明确推迟：

- 隐式乘法，如 `2(3+4)`
- 特殊字符相量输入，如 `5∠30`
- 用户函数落盘
- 矩阵和向量
- 单位系统
- 自然语言输入

## 10. 下一步建议

完成蓝图后，下一步建议先进入“统一数据模型”模块的详细规格设计。该阶段应产出：

- 所有核心类型字段表
- AST 节点定义
- 错误对象定义
- 上下文对象定义
- 函数定义结构

等这一层稳定后，再进入 Tokenizer 和 Parser。不要先写 CLI，更不要先写网页。
